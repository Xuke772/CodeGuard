# SPEC_PROCESS.md — 规约与计划过程文档

## 一、Brainstorming 关键节点

### 1.1 初始构想

项目命题：**构建一个 Coding Agent Harness**。核心洞察来自作业文档："Agent = LLM + Harness"，LLM 只是 CPU，工程价值在于 harness 层。我的初始想法是做一个"能安全编码的 AI 助手"，但具体形态模糊。

### 1.2 AI 追问的关键问题及我的决策

Superpowers 的 brainstorming 技能依次追问了以下问题，每个问题都推动设计更清晰一步：

**Q1: 编程语言？**
- AI 追问了语言选型对生态、分发、测试的影响
- 我选择 Python，因为 AI 生态最成熟，OpenAI SDK 直接兼容 DeepSeek，pytest 测试生态完善

**Q2: LLM 供应商？**
- AI 提出了 OpenAI / Anthropic / 国内模型 / 多供应商四种方案
- 我选择 DeepSeek / 国内模型，因为国内可直接访问，兼容 OpenAI API 格式，价格低廉

**Q3: 重点深入维度？**
- 这是最关键的设计决策。作业要求六个维度都有最低实现，但选一个深入
- AI 推荐治理（护栏/沙箱/HITL），理由是"天然由代码构成，最能体现工程深度"
- 我采纳了这个建议，因为治理确实是最适合用确定性代码实现、最能用 mock 测试验证的维度

**Q4: 分发形态？**
- AI 推荐 Docker 容器，理由是 Python 项目最常用、CI 友好
- 我选择 Docker

**Q5: WebUI 方案？**
- 作业要求提供可访问的 WebUI，但 harness 本质是 CLI 工具
- AI 推荐 FastAPI + 轻量前端，理由是 Python 生态契合、足够轻量
- 我采纳

**Q6: 任务范围？**
- AI 追问 harness 面向什么规模的 coding 任务
- 我选择通用编程任务，因为工具集需要足够通用

### 1.3 方案选择

AI 提出了两种方案：
- **方案 A（我选择）**：精简内核 + 深度治理，三级护栏 + HITL 状态机 + 沙箱执行
- **方案 B（否决）**：插件化架构 + 均衡发展

否决方案 B 的理由：插件抽象层消耗大量工作量，六个维度都浅尝辄止，不符合作业"深度优先"要求。

## 二、关键迭代记录

### 第一轮：功能规约细化

**AI 呈现**：分模块描述功能规约——Agent 主循环、工具层、治理层、反馈闭环、记忆层、配置层。

**我的反馈**：确认所有模块设计，特别关注治理层的三级分类（SAFE / HITL_REQUIRED / BLOCK）和 HITL 状态机。

**修订**：无重大修订，但强调了反馈闭环的"最多 3 轮修正"和"标记为 NEEDS_HUMAN"的边界条件。

### 第二轮：系统架构

**AI 呈现**：组件图 + 数据模型表 + 数据流描述。

**我的反馈**：确认架构清晰，但要求补充外部依赖的替代方案。

**修订**：在架构中加入了 ripgrep 的纯 Python 回退方案和 keyring 的环境变量回退方案。

### 第三轮：领域与机制设计

**AI 呈现**：Coding 领域的四类机制分析 + 治理维度的详细实现方案。

**我的反馈**：确认治理作为重点维度，要求每个子机制都有明确的 mock 可测试性说明。

**修订**：补充了"危险命令分类"、"HITL 状态机"、"文件路径安全"、"沙箱执行"四个子机制的 mock 可测试性对照表。

## 四、对 Brainstorming 技能的反思

### 做得好的地方

1. **强制追问模糊点**：brainstorming 不会让含糊的设计通过，每个关键决策（语言、供应商、重点维度）都被追问到明确
2. **方案对比**：提出 2-3 种方案并给出推荐理由，让决策有依据而非拍脑袋
3. **逐段确认**：设计分节呈现，每节确认后再继续，避免后期大改
4. **硬性门禁**：禁止在 SPEC 和 PLAN 完成前写任何代码，这个纪律非常有效

### 不足之处

1. **问题粒度有时过大**：例如"WebUI 方案"这个问题，AI 只给了 4 个选项，但实际应该先讨论"是否需要 WebUI"再讨论"用什么框架"
2. **缺乏对"最低实现"的指导**：六个维度的最低实现程度没有明确标准，可能导致实现时理解偏差
3. **对凭据安全的追问不够深入**：只是确认了 keyring 方案，没有追问"首次录入流程"、"查看状态的安全显示"等细节——这些在 SPEC 中补充了，但 brainstorming 阶段应该更早触及

## 五、冷启动自我验证

### 验证方法

按照作业要求 §4.5，使用一个**与主开发智能体（OpenCode）不同的 agent**，在**全新 session** 中，仅凭 `SPEC.md` + `PLAN.md`，不提供任何对话历史，尝试实现 1-2 个 task。

### 验证过程

选择了一个不同的 agent 实现 PLAN 中的 Task 2（Config Module）和 Task 6（Governance Module）。

**Task 2 执行情况**：
- 智能体按 SPEC 和 PLAN 的指示，成功创建了 `config.py` 和 `test_config.py`
- 中途暂停 1 次提问：`GuardrailRule` 的 `risk_level` 字段应该用什么类型？—— 暴露了 SPEC 中数据模型表未明确标注字段类型
- 产出与预期一致，测试通过

**Task 6 执行情况**：
- 智能体按 PLAN 步骤创建了 `governance.py` 和 `test_governance.py`
- 中途暂停 2 次提问：
  1. `SENSITIVE_FILES` 中的 `*.pem` 和 `*.key` 应该怎么匹配？—— 暴露了 SPEC 中敏感文件匹配规则不明确
  2. HITL 状态机在 `APPROVED` 状态后如何回到 `IDLE`？—— 暴露了状态转换图中缺少 `reset()` 的描述

### 暴露的 SPEC 缺陷

| 缺陷 | 严重程度 | 修订 |
|------|---------|------|
| 数据模型未标注字段类型 | 中 | 在 SPEC §7 数据模型表中增加"类型"列 |
| 敏感文件匹配规则不明确 | 高 | 在 SPEC §3.3 治理层中补充"敏感文件匹配使用文件名精确匹配和扩展名后缀匹配" |
| HITL 状态机缺少 reset 路径 | 中 | 在 SPEC §3.3 治理层状态机图中增加 reset 转换 |

### 关键发现

1. **SPEC 的清晰度被高估了**：我和主 agent 在 brainstorming 中沉淀了大量共享上下文（如"为什么用 keyring 而不是环境变量"、"为什么护栏是正则而非 ML"），但这些在 SPEC 中并未明文写清。新 agent 在每一处未写明的假设处都会受阻。

2. **PLAN 的代码示例是双刃剑**：PLAN 中的完整代码示例帮助新 agent 快速理解意图，但也让 agent 倾向于照抄而非理解。当代码示例与 SPEC 描述不一致时（如敏感文件匹配），agent 会困惑。

3. **最有价值的反馈是"新 agent 在哪里暂停"**：暂停点精确标记了 SPEC 中最模糊的地方，这是在单人项目中无法从"同侪评审"中获得的最宝贵的客观信号。

### 对 SPEC 的修订

修订前关键差异：
- 数据模型表：`| 实体 | 字段 | 说明 |` → 增加 `| 类型 |` 列
- 治理层：`SENSITIVE_FILES = [...]` → 增加"匹配规则：文件名精确匹配，`*.` 前缀表示扩展名匹配"
- HITL 状态机图：`APPROVED/DENIED → IDLE` → 增加 `reset()` 显式转换
